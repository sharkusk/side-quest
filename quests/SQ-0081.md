--- note 2026-07-05T21:25:59Z ---

CONFIRMED broken on Windows (2026-07-05, user release-testing v0.1.3): the plugin MCP server fails to start on Windows and the binary is never provisioned. Strong evidence that Claude Code does NOT resolve the extensionless plugin.json command `${CLAUDE_PLUGIN_ROOT}/bin/side-quest` to the sibling `bin/side-quest.cmd` on the Windows MCP spawn (i.e. it is not going through cmd.exe/PATHEXT). So this is no longer a "verify" — it is a real defect: the MCP command needs a Windows-resolvable form. Open design question being researched: does mcpServers support per-OS command selection (so one declaration can be `bin/side-quest` on POSIX and `bin/side-quest.cmd` on Windows), or is a different pattern required? Related but distinct from the POSIX-shim-download-under-Git-Bash bug (separate quest). Reclassify priority up once the fix path is known.

--- note 2026-07-05T21:30:48Z ---

Root cause + fix options, from authoritative Claude Code docs (claude-code-guide agent, 2026-07-05):\n- Claude spawns a plugin MCP `command` via Node child_process.spawn() with NO shell, so it does NOT honor Windows PATHEXT. An extensionless `${CLAUDE_PLUGIN_ROOT}/bin/side-quest` fails on Windows (ENOENT). POSIX works only because spawn honors the shebang there. (Refs: node spawn PATHEXT issue; CC Windows MCP guides requiring `cmd /c`.)\n- mcpServers has NO per-OS field (no windows/osx/linux key) in plugin.json or .mcp.json. Confirmed against plugins-reference + mcp.md.\n- Bash TOOL on Windows = Git Bash (CC setup docs, CLAUDE_CODE_GIT_BASH_PATH). So user-typed `side-quest` hits the extensionless POSIX shim (SQ-0082); the MCP server is a separate Node spawn (this quest).\n\nSo ONE mcpServers command must spawn (shell-less) on BOTH OSes. Candidates: `node` (reliable — CC is Node, its docs use command:node/npx for MCP servers) > `bash`/`sh` (Git Bash not guaranteed on the MCP-spawn PATH on Windows). Recommended fix: command:"node", args:["${CLAUDE_PLUGIN_ROOT}/bin/launch.js","serve"]; launch.js does process.platform detection and execs bin/side-quest (posix) or bin/side-quest.cmd (windows), which keep the existing per-OS provisioning. Tradeoff: adds a small JS launcher to a pure-Go plugin (acceptable since node is always present under Claude Code). Alternative: defer Windows plugin, document manual-setup (install binary + project .mcp.json) as the Windows path for now. Decision pending user.

--- note 2026-07-06T01:54:46Z ---

Resolved by SQ-0089 (shipped v0.1.6). The node-launcher fix this quest landed on (command:node + bin/launch.js) was implemented, released, then SUPERSEDED: on a native-installer Windows box there is no node on PATH, so the MCP spawn ENOENTed and failed -32000. SQ-0089 points the MCP command straight at the provisioned native binary (${CLAUDE_PLUGIN_DATA}/bin/side-quest.exe), validated connecting on the real Windows machine. Closing done — the Windows MCP-command-resolution problem this quest tracked is fixed.
