--- note 2026-07-03T20:10:01Z ---

Investigated a standalone pre-push hook approach and found it unworkable: git EXCLUDES a non-fast-forward ref from the pre-push hook's stdin. Verified empirically — for a diverged quest-ref push the hook fires (argv present) but stdin is empty, while a fast-forward push lists the ref. So a pre-push hook fundamentally cannot see the very rejection SQ-0032 is about. Reliable, network-free detection needs the remote's value already on disk, i.e. a remote-tracking ref updated by fetch (e.g. refs/side-quest-remote/quests via an added fetch refspec) — exactly the tracking-ref infrastructure SQ-0031 (sync) introduces. Alternatives are all poor: per-push `git ls-remote` (network latency every push, and still no object for merge-base), or a sha-only compare (false-positives on every normal ahead-of-remote push). DECISION: fold SQ-0032 into the SQ-0031 sync design — once sync maintains a tracking ref, a pre-push (or pre-sync) check against it is a few lines and reliable. Reverted the exploratory pre-push code; nothing shipped for SQ-0032 yet.
