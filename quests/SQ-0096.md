PLAN. In cmdOnboard (cmd/side-quest/onboard.go), after step 1 (s.Init()) and step 2 (cmdInstallHooks, which configures the origin fetch refspec via addRefspec), add a step: if an 'origin' remote is configured, run a sync to pull the remote's existing quests into the local ref BEFORE the user creates any. store.Sync already fetches refs/side-quest/quests explicitly into the tracking ref and three-way-merges into the live ref (no-common-ancestor base handled), and isMissingRemoteRef makes a remote with no quest ref yet a clean no-op. So: detect origin (git remote get-url origin), and if present call s.Sync(remote, opts) best-effort, printing a short 'pulled N existing quests' / 'already up to date' line; no origin => skip silently. This makes a fresh clone start from the remote's current quests so sequential IDs continue correctly and the first push has a shared merge base. Complements SQ-0030 (random-id default on clones) and SQ-0092 (docs). Note: id_strategy=random remains the robust fix for concurrent OFFLINE creation across clones; onboard-sync closes the fresh-clone window but two clones creating quests offline before syncing can still collide under sequential ids — so the belt-and-suspenders answer is onboard-sync AND defaulting clones to random ids. Add a test: onboard in a clone whose origin has quests ends with those quests present locally and the next new id continuing past them.
