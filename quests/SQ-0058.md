--- note 2026-07-04T16:15:16Z ---

Cross-project evidence (2026-07-04): the .gitignore shim block is HAND-maintained — side-quest writes no .gitignore at all. The user's OTHER project already carries the full four-entry block including /.githooks/pre-push and the "these four" comment, copied from side-quest's own block and manually corrected, while side-quest's own repo still had the three-entry SQ-0057 bug until today. So the dogfood repo lagged the user's manual fix, and the user has been patching this per-project by hand each time a shim appears. The hand-written marker-style comment block is effectively a prototype of the feature: install-hooks should write/refresh a marker-guarded, version-aware gitignore block listing its shims whenever the resolved hooks dir is inside the work tree (tracked hooksPath). Scope = the FULL auto-managed block (warning-only is insufficient — user is still copy-pasting). Since the user habitually uses a tracked .githooks layout, this is their normal case, not an edge case → recommend launch=0.1.0.

--- note 2026-07-04T16:24:07Z ---

Root-cause / design fork (from hooks.go:45-46): the shims hardcode the ABSOLUTE binary path deliberately — "so the hooks always run the exact side-quest that installed them (no PATH reliance)" (robust when git is launched by a GUI client or cron that doesn't inherit shell PATH). That absolute path is the ONLY reason the shims are machine-local and thus leak in a tracked core.hooksPath. So there are two coherent designs, and SQ-0058 should pick one, not assume (A):
  A (today): absolute-path shims → machine-local → auto-manage a marker-guarded .gitignore block (this quest's original scope).
  B (alternative): PATH-relative shims (bare `side-quest ...`) → byte-identical across machines → can be TRACKED/committed alongside the repo's own hooks (e.g. the pre-commit doc-guard) → NO gitignore needed at all. Cost: breaks if git runs without side-quest on PATH (GUI git clients, cron).
Decision hinges on whether GUI/cron git invocation is a supported case. If not, B dissolves the problem instead of managing it. Revisit before implementing A.

--- note 2026-07-04T16:36:17Z ---

DECISIVE against option A — composing into a TRACKED hook (verified in installOneHook, hooks.go:197-232): when a repo already has an existing post-commit (or any hook) under a POSIX shell, install-hooks APPENDS its marker block — with the absolute machine-local path — into that file (hookComposed). If the hook is tracked, .gitignore CANNOT help (you can't ignore a modification to a tracked file). The user is then forced to either commit the absolute path (pollutes the shared hook, breaks it for every teammate whose binary path differs) or keep a permanently dirty tree (one `git add -A` from leaking it; a stash/checkout can silently wipe the sq block and disable linking). Non-sh hooks are skipped (hookSkipped) — user integrates by hand. Conclusion: SQ-0058 option A (auto-gitignore) covers only the NEW-separate-shim case; it does nothing for the composed-into-tracked-hook case. Only design B (PATH-relative shims) makes the appended line committable into a shared hook. This makes B the correct direction, not merely the tidier one. Consider splitting the composed-tracked-hook corruption into its own bug if we keep A for any reason.
